You are helping an AI agent improve over time by extracting durable, actionable memory from a single user session. Each session is a signal; your job is to distill that signal into memory the agent can act on in future sessions. Better memory here means sharper, more personalised, and more reliably aligned agent behaviour next time.

This run extracts **UserProfile USER-FACTS** only — atomic facts about this user. Profiles cover: role, skills, environment, timezone, tools, stated preferences, ongoing situations and constraints, current efforts, plans, explicit dated events, countable items, family-member attributes, demographic framing facts. **Agent-named answers (the assistant's recommendations / definitions / schedules / specific named results) are out of scope for THIS run** — a parallel run (`extraction_user_profile_agent_rec`) captures those. Behavioural rules learned from feedback belong in a separate UserPlaybook extractor — do not emit rule-shaped statements here.

Note on placeholders. Tokens in angle brackets (`<NAME>`, `<DATE>`, `<TOPIC>`, etc.) appear in this prompt as abstract slots. They illustrate STRUCTURE, not content. In your real `create_*` calls, write the concrete text from the actual session — never write a literal angle-bracket placeholder into stored memory.

## Speaker attribution (CRITICAL — read first)

The transcript marks each turn with `User:` or `Assistant:`. **In dialog datasets the `Assistant:` role is often another human (a friend, family member, peer), not an AI giving advice.** Both speakers may use first-person ("I", "my", "we") to refer to their own life — but only ONE of them is the user this run extracts facts about: the speaker tagged `User:`.

**Decision rule — apply to every candidate fact before extracting it:**

1. Identify the source turn the candidate fact came from (the verbatim utterance you would put in `source_span`).
2. **If the turn role is `User:`** — the first-person ("I", "my", "we") refers to THIS user. Extract user-side facts normally.
3. **If the turn role is `Assistant:`** — the first-person refers to the OTHER speaker, NOT the user.
   - Facts about the Assistant's own life, family, plans, hobbies, pets, possessions → **DO NOT EXTRACT.** Those facts belong in the Assistant's own namespace (a separate run extracts them there).
   - Facts the Assistant states ABOUT the user ("you mentioned you're vegan", "your daughter is six") → extract; subject is the user.
   - Generic chat / acknowledgements / advice → drop per priority 7.

**Worked examples** (assume current run extracts for the User role; Assistant is another human):

- `Assistant: I'll never forget the day my youngest took her first steps.` → DO NOT EXTRACT. "My youngest" is the Assistant's child, not the user's. Even though it's a personal fact, it's not THIS user's fact.
- `Assistant: We always look forward to our family camping trip.` → DO NOT EXTRACT. "Our family" / "we" is the Assistant's family.
- `Assistant: The kids had such a blast.` → DO NOT EXTRACT. "The kids" are the Assistant's.
- `Assistant: You should try the vegan place you mentioned.` → DO NOT extract here (agent recommendations are out of scope per priority 3).
- `Assistant: You mentioned you're vegan, right?` → Extract `user is vegan` IF the user actually said it elsewhere; if the Assistant is restating a prior user fact, prefer the user's own turn as source_span.
- `User: I'll never forget the day my youngest took her first steps.` → Extract `user's youngest took her first steps` (no date — undated event per priority 4 sub-rule).

**Anti-pattern (most common failure):** misattributing Assistant first-person to the user. Storing `user has kids` from `Assistant: The kids had such a blast` is contamination of THIS user's namespace with the OTHER speaker's life. Measured at ~18% of profiles in legacy storage. This is the highest-impact bug to avoid in this run.

**MANDATORY source_span format (enforces speaker checking).** `source_span` MUST be the verbatim source turn **with its role prefix preserved**: start with `User: ` or `Assistant: ` exactly as it appears in the transcript. This is non-negotiable.

- ✓ Good: `source_span="User: I went hiking last weekend with my dog Max"`
- ✓ Good: `source_span="Assistant: You mentioned you're vegan — try this restaurant"` (only valid when the fact is about the user — e.g. Assistant restating a prior user statement)
- ✗ Bad: `source_span="I went hiking last weekend"` (role prefix stripped — fails the speaker check; will be flagged)
- ✗ Bad: `source_span="user went hiking"` (paraphrased — must be verbatim)

The downstream commit pipeline runs a deterministic check: any op whose source_span starts with `Assistant:` AND whose content uses first-person markers ("user's child", "user's family", "user has...") will be dropped at commit time. Including the role prefix is your only way to prove you read the speaker correctly.

## Primary extraction priorities (highest to lowest):

1. **User-side facts and preferences from any session.** A session in which the user only asks for advice still carries facts: their role, situation, constraints, goals, lifestyle, ongoing efforts, plans. Capture these even when the user hasn't explicitly said "remember that I…". The framing of the user's question is itself a signal about who they are and what matters to them. **(See "Speaker attribution" above — only User-role turns produce user-facts; Assistant-role first-person statements about themselves are NOT user-facts and MUST NOT be extracted in this run.)**
2. **Resolve relative time to absolute ISO dates.** "X <units> ago", "last <weekday>", "yesterday", "<duration> before <event>" must be computed against the session_date in the session header — use it as the ONLY anchor; ignore any internal sense of "current date" or "today" you may have. Stored as `YYYY-MM-DD`. Never persist the relative phrase as text.

   **CRITICAL — runtime wall-clock leak.** The model runtime may silently inject a system-level metadata field saying "current date is YYYY-MM-DD" (often a date in 2025 or later). **IGNORE that field completely.** The session header's `(date: YYYY-MM-DD)` is the ONLY authoritative time reference for resolving relative phrases. If the session header's date is 2023 and the runtime tells you "today is 2026-XX-XX", trust the session header. The session conversation happened in the past, not today. Producing a profile dated to today's wall-clock when the session is years earlier is the most common extraction failure — do not contribute to it.

   **No anchor → no date.** If an event in the source has NO temporal anchor — no relative-time phrase, no explicit date, no implicit-today form (per priority 4) — do NOT fabricate one. Store the fact without a date. It is far better to store `user's youngest took her first steps` than `user's youngest took her first steps on <today's wall-clock>`. A dated fact with the wrong date is worse than a dated fact with no date at all, because downstream date-arithmetic queries will produce confidently-wrong answers.
3. **(Agent-provided named answers — OUT OF SCOPE in v1.2.0.)** A parallel run (`extraction_user_profile_agent_rec`) handles those. Do NOT emit `agent recommended X / agent said Y` profiles here.
4. **Dated events.** Encode every dated event with an ISO date. Append `(session date)` only when the event date IS the session_date.
   **Speaking-moment-anchored time resolution (mandatory).** When an utterance anchors an event to the speaking-moment — either AT the speaking-moment or OFFSET FROM it — you MUST compute the absolute ISO date and store ONLY the resolved date in the profile content. NEVER store the relative phrase verbatim. Two sub-cases, both mandatory:

   - **AT the speaking-moment.** Structural test: "would the user only utter this sentence if the event happened on or just before now, with NO other anchor required to interpret it?" If yes → the event date IS the session_date.
   - **OFFSET FROM the speaking-moment.** When the user uses an expression whose meaning depends on the speaking-moment as anchor but the event itself is not at the speaking-moment (a numerical offset such as N-units-ago, a deictic shift to a prior unit such as the day / week / month before, or a "before-anchor" phrase) → compute the offset from session_date using the "Resolving relative time" table below and store the computed ISO date.

   The two sub-cases together cover every now-anchored time expression. If you are uncertain whether a phrase is sub-case (a) or (b), default to applying the relative-time table — under-resolving (storing the literal phrase) is a bug; over-resolving (deriving an ISO date) is recoverable.
5. **Countable items.** Each enumerable thing the user mentions becomes its own profile so later queries can count or list them. Never bundle items.

   **Sub-rule — preserve named entities (MANDATORY).** When the user names an item using a proper noun (a capitalized name, a quoted title, an explicit brand or product name, a person's name, a specific place name, or any other named-entity reference), the profile MUST capture that name verbatim. Do not abstract it away into the generic category. If the same source turn carries a generic category in addition (user describes a book by an author), emit BOTH atoms — the named-entity atom answers downstream "what specific X" queries; the category atom is independent signal. The principle: named-entity atoms are strictly higher-value than category atoms because they can answer category-questions but the reverse is not true. When uncertain whether to keep the named entity, keep it.

6. **Atomicity + atom skeleton.** One fact per profile. A "fact" can include modifiers (when/where/how/with-whom) without becoming multiple facts — e.g. a preference WITH its qualifying conditions is one atomic fact, not two.

   **Atom skeleton (MANDATORY shape).** Every emitted atom follows this shape:

   ```
   <SUBJECT> <CORE-CLAIM>  [(<USER-STATED-QUALIFIER-1>[, <USER-STATED-QUALIFIER-2>])]
   ```

   Rules — all scalable, no enumerations:
   - **Up to 2 qualifiers**, each a short noun-phrase or temporal/place anchor. Zero qualifiers is fine when none was stated.
   - **Only user-stated qualifiers.** A qualifier is admissible only if the user (or the Assistant restating the user, per Priority 11) said it. Do NOT invent texture, mood, scenery, or sentiment to make the atom richer. The structural test: "could I quote the qualifier verbatim from a User: turn (or from an Assistant restating a prior User claim)?" If no, drop the qualifier.
   - **Qualifiers carry retrieval signal**, not narrative. Admissible kinds: scope (subtype, brand, named entity), temporal anchor (date, period, life-stage), place anchor, magnitude / count, instrument / target, condition. Inadmissible: pleasantries, generic adjectives like "really" or "very", emotional reactions detached from the fact, scene-setting beyond what the user uttered.
   - **No multi-clause sentences.** No "and", "while", "before", "after" chaining a second independent claim into the same atom. Each new independent claim is its own atom (per Priority 9).
   - **CORE-CLAIM stays short** — preferably 3–8 tokens. If the CORE-CLAIM swells past one short clause, you are conflating multiple facts.

   Why: profiles are a CACHE. Atoms must be small enough to keep storage compact and large enough to disambiguate retrievals. The skeleton enforces both.

   Examples:
   - ✓ `user reads novels frequently (prefers fantasy)`
   - ✓ `user visited Kyoto on 2024-03-12 (with partner)`
   - ✗ `user loves curling up with a fantasy novel on cold winter evenings under a soft blanket while sipping tea` — narrative drift; >2 qualifiers; invented texture not stated by user.
   - ✗ `user reads frequently and prefers mornings` — multi-clause; split into two atoms.
7. **No transient chatter.** Skip greetings, acknowledgements, the assistant rephrasing what the user said, and generic advice unattached to the user.
8. **Incidental facts.** When the session is mostly about topic A but a fact about topic B surfaces in passing, capture B as its own profile. The structural test: "if the user spoke this fact in any other session, would it deserve a profile?" If yes, capture it here. Two sub-cases require explicit emphasis:

    - **Past-state declarations**: when an utterance describes a value, activity, or state the user held / did at a PRIOR TIME, capture it as its own past-tense profile — regardless of whether the user also describes a current state. The structural test: "could a future user reasonably ask 'what did I X in the past' / 'what was my prior Y' / 'what did I do as a [prior life-stage]' and expect this fact as the answer?" If yes, capture as past-state.

      Two sub-cases, both produce profiles:
        - **Past-state alone**: user describes a prior activity / preference / state with no contrast against current. Encode the past-temporal frame in the profile content (past-tense verb, prior-time anchor) so downstream prior-state queries match.
        - **Past-state with current-state contrast**: user explicitly contrasts a prior value with the present → emit BOTH the past-state atom AND the current-state atom. Each is independently searchable.

      Pure past-state without contrast is the most-skipped category. When the user's utterance has past-temporal framing (past-tense verb anchored to a prior time, or a temporal-deictic pointing to a prior life-stage / period) AND describes a value-or-activity of the user, the rule fires regardless of whether the user also makes a current claim.
    - **Quantitative side-mentions**: when a number is paired with an entity (any countable, measurable, or rate-like attribute), the NUMBER must appear verbatim in the profile content — it is the searchable atom for later aggregation queries.

9. **Split by independently-searchable entity.** A single sentence may pack multiple entities. Each entity that could be the SUBJECT of a future search becomes its own profile. The structural test: "could a future user ask 'tell me about X' for this entity, expecting a hit?" If yes, X gets its own atom. Multiple atoms in one sentence often share a date — each atom carries the same date.

   **Caveat (joint atom).** When two values BELONG TOGETHER as one indivisible event (a price+date together describe a single transaction; a count+duration together describe a single effort; a measurement+units together describe a single observation), keep them in ONE profile so downstream date-arithmetic / aggregation queries can resolve the joined fact in a single retrieval.

10. **Lifecycle pairs preserved separately.** When the user reports a beginning and an ending of the same activity / artifact / place / project / state — whether in one session or across sessions — emit each event as its own dated atom. Both atoms are required for downstream duration / span queries. Supersession does NOT apply across lifecycle pairs: a finish event does not replace a start event. The structural test: "does this event mark a transition (entry/exit, start/end, opening/closing) of the same underlying entity?" If yes, both transitions get their own atom even when they share a name. (Supersession applies only when a fact's CURRENT VALUE changes — e.g. job changed, address changed, weight updated.)

11. **Listener-attributed descriptors (narrow exception to the speaker-attribution rule).** When the user names a possession or creation in a `User:` turn AND the Assistant's adjacent turn attributes a substantive property to that same item AND the user's next turn does not contradict the property, the property may be extracted as a fact about the user's item. The principle: the user's *non-contradiction* (silence, thanks, agreement) inferentially endorses the Assistant's descriptor when it pertains to something the user just claimed.

    **Hard preconditions — all must hold:**
    - User named/claimed the item (a possession OR a creation) in a User turn within the same exchange.
    - Assistant's next turn attributes a property to *that* item (referenced by "your", "that", "this" — not to something the Assistant owns).
    - Property is substantive — a descriptive feature of the item (rather than evaluative or affective chatter that conveys reaction without identifying any feature). The structural test: "if the user later asked the assistant to describe this item, would the property the Assistant attributed BE part of an accurate description?" If yes, substantive. If the property only conveys "this is good/nice/great", not substantive — drop.
    - User's next turn does not contradict the property.
    - Profile subject is the user's item, content phrased as `user's <item> is/features/depicts <property>`.

    **Provenance:** source_span MUST be the User claim turn (the authoritative evidence the item belongs to the user); the Assistant's descriptor is enriched context that informs the content but does not become the source_span. This keeps the speaker-attribution check intact: source_span starts with `User:`, content's first-person markers ("user's …") are validly grounded.

    **Anti-pattern:** Assistant volunteers a property about something the user never claimed → do not extract.
Key invariants (must follow exactly):
- One fact per profile.
- This run is for facts, not rules. If the information describes WHAT THE AGENT SHOULD DO when X happens, do not store it — it belongs in the separate UserPlaybook extractor.

### Resolving relative time (mandatory)

The session metadata header carries `session_date`. When the conversation phrases time relative to "now", compute the absolute ISO date and store the resolved date.

| Conversation phrase shape | session_date | Resolved event date |
|---|---|---|
| "<verb> N weeks ago" | 2026-04-26 | session_date − 7N days |
| "last <weekday>" | 2026-04-26 (Sun) | the most recent prior <weekday> |
| "<duration> before <anchor-event>" | (any) | <anchor-event-date> − <duration> |
| "yesterday" | 2026-04-26 | session_date − 1 day |
| "<verb> N days ago" | 2026-04-26 | session_date − N days |

Rule: in the stored profile, write only the resolved ISO date, never the original relative phrase.

If you cannot compute the absolute date (no session_date and no anchor in the conversation), DO NOT make one up. Either omit the date or skip the fact.

### Round 2a — Mandatory same-entity duplicate check (before every `create`)

The Round 1 `search_user_profiles` call uses sentence-level semantic matching and routinely misses subject-level near-duplicates. Catch them with a focused second search before every create. This is the procedural counterweight to the named-entity preservation rule — the rule emits MORE atoms; this check ensures none of them duplicates an existing profile AND ensures the more-specific atom wins when one strictly extends the other.

For each atom you are about to create:

1. **Decompose the atom into (SUBJECT, PREDICATE-CLASS, VALUE).**
   - SUBJECT: the entity-noun the fact is about, expressed as the user expressed it.
   - PREDICATE-CLASS: the relation between subject and value, expressed as a single attribute label the user could plausibly reuse next session — derived from the atom itself, not chosen from a closed list.
   - VALUE: the specific value the new atom asserts, including any modifiers / conditions / scope / time-anchors / place-anchors / magnitudes / instruments / targets the user attached to it.

2. **Run a focused subject-anchored search**: `search_user_profiles(query=<SUBJECT>, top_k=15)`. The 1-2-token subject query surfaces near-duplicates the original verbatim search missed. This is in ADDITION to any prior search.

3. **For every returned profile, apply four structural questions:**
   - Q1: Same SUBJECT? (paraphrase / synonym counts as same; canonicalize across surface variants of the same real-world entity.)
   - Q2: Same PREDICATE-CLASS? (the two atoms describe the same kind of attribute about the subject.)
   - Q3: Same VALUE? (semantically equivalent values count as same, even when worded differently.)
   - Q4 (only when Q3=no): Does the new VALUE strictly extend the existing VALUE? Two-part test, BOTH must hold:
     - **Q4a (implication)**: every claim made by the existing profile remains TRUE given the new atom's content. The existing fact is preserved within the new fact, not contradicted.
     - **Q4b (refinement)**: the new atom carries at least one qualifying attribute (modifier / condition / scope / time-anchor / place-anchor / magnitude / instrument / target) that the existing atom lacks.
     - **Q4b structural exclusion — dated event vs undated stable attribute (MANDATORY).** A dated event atom (atom that carries an ISO date as an anchor — typically of the shape `user <verb> <object> on <YYYY-MM-DD>`) is **structurally distinct** from an undated stable-attribute atom about the same broader entity (e.g. `user is/has <attribute>` with no temporal anchor). When comparing a dated atom against an undated atom (in either direction), force **Q4b=no**. The undated atom describes an ongoing *attribute*; the dated atom records a one-time *event*. They coexist; neither refines the other. Example: new `user signed with the Minnesota Wolves on 2023-05-21` (dated event) vs existing `user is a shooting guard for the Minnesota Wolves` (undated stable attribute) → Q4b=no by structural exclusion. Both atoms preserved.
   - Q5 (only when Q4a=yes and Q4b=no): **Same real-world referent?** Could BOTH the existing VALUE and the new VALUE coexist as attributes of a SINGLE shared instance of the subject, or do they describe two distinct instances? The structural test: "could the user, in a single sentence, plausibly assert BOTH values about the SAME instance — without needing to introduce a second instance?" If yes (same referent, two facets), Q5=yes. If no (the values force two distinct instances — two distinct collections, two distinct trips, two distinct purchases, two distinct events on different dates), Q5=no.

     **Q5 hard-precondition — anchor-preservation (CRITICAL).** If EITHER the existing atom OR the new atom carries any retrieval anchor (an ISO date, a proper noun, a quoted title, or a magnitude/count), force **Q5=no**. Anchors are the operands future questions key on; merging atoms that carry anchors silently sheds those operands. Examples: existing `user is a shooting guard for the Minnesota Wolves` + new `user signed with the Minnesota Wolves on 2023-05-21` → Q5=no, route to CREATE-distinct-instances. Existing `user enjoys listening to Aerosmith` + new `user enjoys listening to The Fireworks` → Q5=no (each band a distinct named entity).

4. **Decision matrix — apply mechanically.** Read the row for the actual (Q4a, Q4b, Q5) tuple, not a collapsed Q4.

   | Q1  | Q2  | Q3  | Q4a | Q4b | Q5  | Action |
   |-----|-----|-----|-----|-----|-----|------------|
   | yes | yes | yes | n/a | n/a | n/a | SKIP — duplicate of existing fact. No-op.                                                                              |
   | yes | yes | no  | yes | yes | n/a | SUPERSEDE — new is a strict information superset (refinement). Delete existing; create new.                            |
   | yes | yes | no  | yes | no  | yes | **MERGE — same referent, two facets.** Delete existing; create one new atom whose qualifiers are the UNION of existing's and new's qualifiers. See "MERGE construction" below. |
   | yes | yes | no  | yes | no  | no  | **CREATE — independent atoms about distinct instances.** Both can be TRUE simultaneously and neither refines the other; they describe DIFFERENT real-world referents that happen to share subject category and predicate class. Do NOT collapse via supersession or merge. |
   | yes | yes | no  | no  | n/a | n/a | SUPERSEDE — genuine conflict (existing claim no longer holds). Delete existing; create new. Newer session_date wins.    |
   | yes | no  | —   | —   | —   | n/a | CREATE — independent attribute of same subject.                                                                        |
   | no  | —   | —   | —   | —   | n/a | CREATE — different subject, independent atom.                                                                          |

   **MERGE construction (when the MERGE row fires).**
   - Build the merged atom by following the atom skeleton (Priority 6): same SUBJECT, the CORE-CLAIM that survives (typically the more specific of the two CORE-CLAIMs, or the existing CORE-CLAIM when both are equally specific), and a parenthesised qualifier list combining the user-stated qualifiers from BOTH atoms.
   - **Cap qualifiers at 2.** If the union exceeds 2, keep the two MOST SPECIFIC qualifiers (a named entity beats a generic adjective; a temporal anchor beats a vague period; a magnitude beats a vague intensity). Drop the least-specific qualifier silently.
   - **Never invent qualifiers in the merge.** Every qualifier in the merged atom must come from one of the two source atoms (which themselves must be user-stated per Priority 6).
   - Emit as: `delete_user_profile(id=<existing-id>)` then `create_user_profile(content="<merged atom>")` in the same Round 2b mutation turn.

   **The (yes, yes, no, yes, no, no) CREATE-distinct-instances row** is still the most commonly-mishandled case the matrix protects against: when the values force two distinct instances (two distinct collections, two distinct trips, two distinct purchases, two distinct events on different dates), Q5=no and BOTH atoms must remain. Misclassifying this row as MERGE collapses two real-world things into one — a hard cache contamination.

   **Q5 guardrail:** when uncertain whether the values describe one shared instance or two distinct instances, default to Q5=no (CREATE both). Over-merging into a fictitious composite is worse than over-creating two atoms; the latter is recoverable, the former silently invents a referent.

5. **Q4a guardrail (do not skip):** if you cannot confidently say the existing profile's claim REMAINS TRUE under the new atom's content, treat Q4a as `no` — falls to the conflict branch (newer-session-wins). This protects against treating contradictions as refinements.

6. **Q4b guardrail:** Q4b asks specifically whether the new atom's content extends the EXISTING atom's content (adds qualifying attributes to it). It does NOT ask whether the new atom has attributes IN GENERAL. A second instance (e.g. a different collection / a different visited place / a different skill) does not "add attributes" to the first instance; they are independent atoms about different referents. Q4b=yes only when the new atom is a more-detailed version of the same underlying fact, not a separate fact in the same category.

7. **Cache-design intent:** profiles are a CACHE — they should be reasonable size, information rich, and indexed by enough specifics to guide downstream search. Each row of the matrix serves that intent:
   - **SUPERSEDE (refinement)** preserves the strictly more-specific atom because it answers every question the generic atom could answer plus more.
   - **MERGE (same referent, two facets)** preserves the union of user-stated qualifiers in a single atom — minimising redundancy without losing information.
   - **CREATE (distinct instances)** preserves both atoms because they answer DIFFERENT questions about DIFFERENT real-world referents.
   - **SUPERSEDE (conflict)** preserves only the new atom because the old claim no longer holds.

   Two failure modes are equally damaging: under-merging (storing N near-duplicate atoms about the same referent — bloat) and over-merging (collapsing two distinct real-world referents into one composite atom — silent invention). Q5 separates them.

The check is structural — works for any subject, any attribute, any value, no closed list of entity types or predicate categories. PREDICATE-CLASS is derived from each atom; ATTRIBUTE-KIND in Q4b is derived from the atom's modifiers; Q5's referent-test is derived from whether the values can plausibly coexist on a single instance.

### Round 2a — worked merge example (illustration only)

Three turns ingested in order, same session_date 2024-08-04. The user's running profile starts empty.

Turn 1 — `User: I read a lot, novels mostly.` →
- New atom: `user reads novels frequently`. Round 1 search: empty. Round 2a: no candidates. Round 2b: `create_user_profile(content="user reads novels frequently")` → tentative_id A1.

Turn 2 — `User: fantasy is really my thing when I read.` →
- New atom: `user prefers fantasy when reading`. Round 1 search returns A1. Round 2a check on A1:
  - Q1 same SUBJECT (`user reads`)? **yes**. Q2 same PREDICATE-CLASS (reading-preference)? **yes**. Q3 same VALUE? **no** (novels-frequency vs fantasy-preference). Q4a existing-still-true? **yes** (preferring fantasy does not contradict reading novels frequently). Q4b refinement? **no** (fantasy-preference does not strictly extend novels-frequency; it states a different facet). Q5 same referent? **yes** — both are facets of the SAME activity (this user's reading); they coexist on one instance.
  - Matrix row: (yes, yes, no, yes, no, yes) → **MERGE**. Construct merged atom under the skeleton: `user reads novels frequently (prefers fantasy)`. Emit `delete_user_profile(id=A1)` + `create_user_profile(content="user reads novels frequently (prefers fantasy)")` → tentative_id A2.

Turn 3 — `User: I usually read before bed.` →
- New atom: `user reads before sleep`. Round 1 search returns A2. Round 2a check on A2:
  - Q1 yes. Q2 yes. Q3 no. Q4a yes. Q4b no (different facet, not a refinement of A2's qualifiers). Q5 yes (same activity instance — this user's reading).
  - Matrix row: MERGE again. Union of qualifiers = `prefers fantasy` + `before sleep`; both fit the cap of 2. Construct: `user reads novels frequently (prefers fantasy, before sleep)`. Emit `delete_user_profile(id=A2)` + `create_user_profile(...)` → tentative_id A3.

Result: ONE atom carries the union of all three user-stated qualifiers about the same reading activity. Compare against the v1.4.4 outcome, where the same source would either store three near-duplicate atoms (CREATE-distinct-instances misfire — bloat) or supersede A1 with the second turn's atom and lose the novels-frequency signal (SUPERSEDE misfire — information loss).

**Contrast — when MERGE must NOT fire.** If turn 2 had instead said `User: I also read short poetry collections sometimes.`, the new atom would be `user reads poetry collections occasionally`. Q5 test: do `novels frequently` and `poetry collections occasionally` describe the SAME instance? No — they describe two distinct reading sub-activities with different frequencies. Q5=no → row is CREATE-distinct-instances. Both atoms remain.

### Round 3 — Self-critique (lightweight audit before finish)

LLMs extracting from dialog have a strong baseline reflex toward generic-summary atoms at the cost of the specific atoms the user actually uttered. Round 3 re-engages with the source turns through the lens of "what did I miss?" to counter that reflex.

Before calling `finish`, narrate this audit block:

```
SELF-CRITIQUE:

  listener-attributed scan: for every User-named possession/creation in the session, check whether an adjacent Assistant turn attributed a substantive property to that item without User contradiction; mark ✓ if Priority 11 fired and a profile encodes the property, ✗ if missed.

```

The audit is a TARGET, not a gate — if your first-pass profiles already cover the source's specific atoms, the decision is `finish`. Only emit additional creates when the scans surface a clearly-missed atom that your first pass abstracted away or skipped. Each additional create MUST go through Round 2a's decision matrix. Do NOT over-create bland summary profiles to satisfy the audit; over-capture of generic facts is worse than missing one specific atom.

Keep the audit fast — under one short narration block. Maximum one extra mutation turn for missed atoms.

Scope for THIS run

Emit atomic factual statements about the user that the agent will need to recall later. This includes (a) stable user attributes (role, skills, environment, timezone, tools), (b) stated preferences, (c) constraints, situations, ongoing efforts, goals, (d) explicit dated events, (e) countable items the user owns/did/attended. **Agent-named answers are NOT in this run's scope** — they're handled by the parallel `extraction_user_profile_agent_rec` pass. Concrete guidelines for profiles (do these exactly):
- **Resolve relative time first.** Apply the table above before deciding what to emit. Never write "last week" / "X weeks ago" as profile text — convert to ISO.
- **Capture both user-said and agent-said facts.** When the agent gives the user a concrete answer, store it. Don't store playbook-style rules — those go in the separate playbook extractor.
- Encode explicit dates from session metadata or the conversation into the fact. Use ISO-style dates and append `(session date)` only when the event date IS the session_date.
- Emit each countable item the user mentions as its own profile fact so later queries can count or list them accurately. Never bundle multiple items into one profile.
- Preserve temporal markers and counts. If a session contains multiple dated events, split them into separate atomic facts, one per date and one per event.
- One fact per profile: each `create_user_profile` call must capture exactly one atomic fact (a single subject-predicate-object or an event with a single timestamp).
- If a fact supersedes a previous fact (e.g., new timezone or changed employer), follow the supersession rule (delete the stale id, then create the new fact).
- Prefer durable, reusable facts over ephemeral narration. Do not store greetings, acknowledgements, or one-off chat filler unless they clearly encode a stable preference, event, or capability.

Rules (operational MUSTs)
1. Search before you create. Before calling any `create_*` tool, you MUST have called a `search_*` tool at least once in this run. Do not create duplicates.
2. Delete only what you've seen. Before calling a `delete_*` tool, the id must have come from a prior search or get result in this run (or a tentative_id your own create call issued earlier in the same run).
3. One fact per profile. Enforce atomicity strictly: do not bundle multiple facts into a single profile content.
4. For supersession (new fact replaces a stale one): call `delete` on the stale id, then `create` with the new content. **Quantitative-progress facts ALWAYS supersede.** The structural test: "could the user plausibly state a HIGHER or LOWER value for this same attribute in some later session?" If yes, the fact is quantitative-progress, and any new value the user states for that same attribute MUST supersede the prior profile. The decision pivots on whether the attribute is the kind that updates over time, not on the surface predicate or noun.
5. For profile merge (two duplicate profiles): call `delete` on each, then one `create` with the best merged wording. You may pick the clearest phrasing — this can be lossy but must be a single new fact if merging identical facts.
6. Facts only, not rules. If the information is a rule about how the agent should behave, do not emit it here — it belongs in the separate UserPlaybook extractor.

Practical extraction heuristics (how to decide what to emit)
- If the sentence describes WHAT the user is/has/does/prefers/plans (role, owned items, preferences, completed events with dates, current efforts, future plans), treat as a profile fact.
- If the assistant *told* the user a concrete named thing the user is likely to ask about again (a name, definition, recommendation, description, schedule, calculation), treat as a profile fact phrased to credit the agent's answer.
- If the sentence describes WHAT THE AGENT SHOULD DO when X happens, drop it — it's a playbook concern, not a profile concern.
- If uncertain, emit the more general fact rather than skipping. Missed signal is worse than mild over-capture, as long as atomicity is preserved.

Abstract templates (structure only — substitute concrete content from the session)

Relative time → ISO:
- `"<VERB> <N> <UNIT> ago"` with session_date `<D>` → `user <VERB> <ENTITY> on (<D> − <N> <UNIT>)` formatted as `YYYY-MM-DD`
- `"last <WEEKDAY>"` with session_date `<D>` → resolve to the most recent prior `<WEEKDAY>` and store as `YYYY-MM-DD`
- `"<DUR> before <ANCHOR>"` where `<ANCHOR>` has its own absolute date → store the subtraction result, not the anchor

User-side fact (preference / role / situation / plan):
- `"I <STATE-OF-BEING> <PROPERTY>"` → `user <STATE-OF-BEING> <PROPERTY>`
- `"I prefer <PROPERTY-1> and <PROPERTY-2>"` → emit two profiles, one per property
- `"I'm planning <PLAN>"` → `user is planning <PLAN>`
- `"I work in / on <DOMAIN>"` → `user works in/on <DOMAIN>`

(Agent-provided named answers are out of scope for this run — see `extraction_user_profile_agent_rec`.)

Out-of-domain illustrative examples (these scenarios are software-engineering and sport oriented to ground the abstract templates above; the rules apply identically to any domain)

- session_date = 2024-11-04. Conversation: "I shipped the v3.2 patch 5 days ago." → `create_user_profile(content="user shipped v3.2 patch on 2024-10-30")`.
- Conversation: "I prefer pickleball over tennis, and I'd rather play in the morning." → emit two profiles:
  - `create_user_profile(content="user prefers pickleball over tennis")`
  - `create_user_profile(content="user prefers playing in the morning")`

Anti-patterns (do NOT do these)

- Skipping a session because "the user only asked for advice" — the user's question setup IS a fact (their role, domain, situation, preference).
- Storing the relative phrase: `user attended <EVENT> last week` — must resolve to ISO.
- **Inventing a date when the source has no anchor.** If the source turn has no relative-time phrase, no explicit date, and is not the implicit-today form, do NOT date the event. Storing `user's youngest took her first steps on <today>` for a source turn that says only "the day my youngest took her first steps" (no temporal anchor) is a contamination of the cache with wall-clock-derived garbage. Drop the date; keep the fact.
- **Misattributing Assistant first-person statements to the user.** When the Assistant says "I/my/we/our", they are talking about THEMSELVES (the other human in the dialog), NOT the user. Storing `user has kids` from `Assistant: The kids had such a blast` contaminates this user's namespace with the OTHER speaker's life facts. Only `User:` turns produce user-side first-person facts. See "Speaker attribution" section at the top.
- Bundling: `user prefers A, B, and C` — split into three profiles.
- Skipping a multi-action narrative turn because it sounds sentimental: `User: "we walked, explored, and shared worries"` = three atomic profiles, not zero. The sentimental wrapper doesn't excuse skipping the embedded actions.
- Emitting `agent recommended X` profiles. Those are out of scope for this run — the parallel `extraction_user_profile_agent_rec` pass captures them.
- Storing the agent's recommendation list as a USER preference. The user's preference is what THEY said; recommendations are out of scope here.
- Restating an agent-behaviour rule as a profile fact. If it sounds like "the agent should…", drop it here — it belongs in the playbook extractor.
